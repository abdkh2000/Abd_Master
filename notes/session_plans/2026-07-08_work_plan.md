## BRINE Code Inspection Checklist

- Verify the exact code folder name on the remote system before documenting it.
- Identify the clean baseline copy and leave it untouched.
- Create a separate working copy for any test case.
- Locate the channel-flow case directory.
- Identify the build scripts, run scripts, and required libraries without running large jobs.
- Inspect `input.inp` for simulation parameters.
- Inspect `ic.inp` for initial-condition settings.
- Inspect `module.F90` for grid-size variables such as `nx`, `ny`, and `nz`.
- Inspect `main.F90` for switches, flags, and buoyancy-related controls.
- Record findings in project notes using placeholders for any private paths or machine-specific details.

- What is the temperature and buoyancy sign convention?

- What nondimensional buoyancy or stratification parameter should be tested first?
- What grid size is appropriate for the first lightweight workflow test?
- Which validation plots are required before enabling buoyancy?
- Which reference DNS datasets should be used for the non-buoyant baseline?
- What output files are essential, and which generated files should stay out of Git?
